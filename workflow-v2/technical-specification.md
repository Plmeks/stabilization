# Техническая спецификация: Рефакторинг приложения управления задачами

## 1. Общее описание

**Цель:** Крупный рефакторинг существующего Next.js приложения для управления задачами команды — переработка логики статусов, упрощение UI, добавление таблицы статистики, обновление дизайна, исправление багов.

**Существующая система:** Next.js, Supabase (PostgreSQL), Jotai state management, 4 таба (QA, Текущие, Выполненные, Статистика).

**Ограничения от клиента:**
- Тесты НЕ требуются
- Минимальная документация

## 2. Use Cases

### UC-1: Отображение задач в QA табе

- **Акторы:** Пользователь, Система
- **Предусловия:** Пользователь авторизован, открыт таб "Новые задачи / QA"
- **Основной сценарий:**
  1. Система загружает ВСЕ задачи из БД (кроме физически удалённых)
  2. Отображает список задач БЕЗ отображения статуса
  3. Для критических задач (priority = 'Авария') отображается визуальный маркер
  4. Задачи с любым статусом (null, "В работе", "Завершена" и т.д.) присутствуют в списке
- **Альтернативные сценарии:**
  - 1a. Нет задач → отображается пустое состояние
- **Постусловия:** Пользователь видит полный список задач
- **Критерии приёмки:**
  - [ ] QA таб показывает ВСЕ задачи независимо от статуса
  - [ ] Статус задачи НЕ отображается в QA табе
  - [ ] Задачи с `priority = 'Авария'` визуально выделены

### UC-2: Создание задачи

- **Акторы:** Пользователь, Система
- **Предусловия:** Пользователь в любом табе
- **Основной сценарий:**
  1. Пользователь нажимает "Создать задачу"
  2. Заполняет форму (название, опционально — флаг "Критическая")
  3. Система создаёт задачу: `status = null`, `priority = null` (или `'Авария'` если отмечена критической)
  4. Задача появляется в QA табе
- **Постусловия:** Задача создана со статусом null
- **Критерии приёмки:**
  - [ ] Новая задача создаётся с `status = null`
  - [ ] Если отмечена "Критическая" → `priority = 'Авария'`
  - [ ] Задача сразу видна в QA табе

### UC-3 (обновлённый): Взятие задачи в работу

- **Акторы:** Пользователь, Система
- **Предусловия:** Задача существует, видна в QA табе
- **Основной сценарий:**
  1. Пользователь нажимает кнопку "Взять в работу"
  2. Система **БЕЗ модалки** выполняет:
     - `status = 'В работе'`
     - `taken_into_work_at = now()`
     - Если `priority = null` → `priority = 'Нормальный'`
     - `assignee` остаётся пустым (НЕ заполняется автоматически)
  3. Задача появляется в табе "Текущие задачи"
  4. Задача остаётся видимой в QA табе
- **Альтернативные сценарии:**
  - 2a. Задача уже имеет статус ≠ null → кнопка "Взять в работу" НЕ отображается
- **Постусловия:** Задача имеет статус "В работе", отображается в обоих табах (QA и Текущие)
- **Критерии приёмки:**
  - [ ] Клик "Взять в работу" — нет модалки, мгновенное действие
  - [ ] `status` меняется на `'В работе'`
  - [ ] `taken_into_work_at` устанавливается
  - [ ] Приоритет = 'Нормальный' если был null
  - [ ] `assignee` всегда остаётся пустым (заполняется только вручную через редактирование)
  - [ ] Задача видна в "Текущие задачи"
  - [ ] Задача НЕ исчезает из QA
- **Удаление кода:** Файл TakeIntoWorkModal удаляется полностью из проекта

### UC-4: Возврат задачи в начальное состояние

- **Акторы:** Пользователь, Система
- **Предусловия:** Задача в табе "Текущие" или "Выполненные", `status ≠ null`
- **Основной сценарий:**
  1. Пользователь нажимает кнопку "Вернуть в QA" (иконка Undo2)
  2. Система сбрасывает следующие поля:
     - `status = null`
     - `taken_into_work_at = null`
     - `assignee = null`
     - `priority` — **НЕ сбрасывается** (остаётся как было)
  3. Задача исчезает из текущего таба (Текущие или Выполненные)
  4. Задача остаётся видимой в QA табе
- **Постусловия:** Задача имеет `status = null`, сброшены связанные поля
- **Критерии приёмки:**
  - [ ] Кнопка "Вернуть в QA" доступна в табах Текущие и Выполненные
  - [ ] После нажатия: `status = null`, `taken_into_work_at = null`, `assignee = null`
  - [ ] `priority` сохраняется (не сбрасывается)
  - [ ] Задача пропадает из таба Текущие/Выполненные
  - [ ] Задача остаётся в QA
  - [ ] Нет ошибки `codes.forEach is not a function` (фикс бага #1)

### UC-5: Фильтрация задач по табам

- **Акторы:** Система
- **Логика фильтрации:**

| Таб | Фильтр |
|-----|--------|
| QA (Новые задачи) | Все задачи (без фильтра по статусу) |
| Текущие задачи | `status IN ('В работе', 'В тесте', 'Блокер')` |
| Выполненные | `status = 'Завершена'` |

- **Критерии приёмки:**
  - [ ] QA показывает все задачи
  - [ ] "Текущие" — только со статусами В работе / В тесте / Блокер
  - [ ] "Выполненные" — только Завершена
  - [ ] Задачи с `status = null` не попадают в Текущие/Выполненные

### UC-6: Фиксация метрик периода

- **Акторы:** Пользователь, Система
- **Предусловия:** Открыт таб "Статистика", период без зафиксированных метрик
- **Основной сценарий:**
  1. Пользователь видит динамически вычисленные метрики для текущего периода
  2. Нажимает кнопку "Зафиксировать метрики"
  3. Система вычисляет метрики по `period_id` (НЕ по диапазону дат):
     - `added_to_backlog` — COUNT задач WHERE `period_id = X` AND `status IS NOT NULL`
     - `added_critical` — COUNT задач WHERE `period_id = X` AND `status IS NOT NULL` AND `priority = 'Авария'`
     - `resolved_total` — COUNT задач WHERE `period_id = X` AND `status = 'Завершена'`
     - `resolved_critical` — COUNT задач WHERE `period_id = X` AND `status = 'Завершена'` AND `priority = 'Авария'`
     - `in_progress` — COUNT задач WHERE `period_id = X` AND `status = 'В работе'`
     - `in_testing` — COUNT задач WHERE `period_id = X` AND `status = 'В тесте'`
  4. INSERT в `period_statistics`
  5. Кнопка "Зафиксировать" заменяется на "Редактировать" (иконка-карандашик)
  6. Метрики теперь берутся из таблицы
- **Важно:** Все метрики фильтруются по `period_id` задачи, а НЕ по date range периода
- **Постусловия:** Запись в `period_statistics` создана
- **Критерии приёмки:**
  - [ ] До фиксации — метрики вычисляются динамически
  - [ ] После фиксации — метрики из `period_statistics`
  - [ ] Кнопка меняется с "Зафиксировать" на "Редактировать"
  - [ ] Фильтрация метрик по `period_id` (не по date range)
  - [ ] `added_critical` использует `priority = 'Авария'`
  - [ ] Запись в БД корректна

### UC-7: Редактирование зафиксированных метрик

- **Акторы:** Пользователь, Система
- **Предусловия:** Период имеет запись в `period_statistics`
- **Основной сценарий:**
  1. Пользователь нажимает "Редактировать" (карандашик)
  2. Модалка с предзаполненными значениями всех 6 метрик
  3. Пользователь меняет значения
  4. Нажимает "Сохранить"
  5. UPDATE в `period_statistics`
  6. Модалка закрывается, метрики обновляются
- **Альтернативные сценарии:**
  - 4a. "Отмена" → модалка закрывается без изменений
- **Критерии приёмки:**
  - [ ] Модалка показывает 6 полей с текущими значениями
  - [ ] Поля числовые, редактируемые
  - [ ] "Сохранить" → UPDATE
  - [ ] "Отмена" → без изменений

## 4. Ограничения и допущения

**Ограничения:**
- Стек: Next.js + Supabase + Jotai — без изменений
- Тесты не пишутся
- Допустимые статусы: `null`, `'В работе'`, `'В тесте'`, `'Завершена'`, `'Блокер'`
- Допустимые приоритеты: `null`, `'Нормальный'`, `'Авария'`

**Допущения:**
- Таблица `periods` существует в БД
- Файл TakeIntoWorkModal удаляется полностью (не комментируется)
- Поле `metrics_snapshot` из таблицы `periods` удаляется
- `assignee` при "Взять в работу" всегда остаётся пустым
- Ошибки chrome extension (баг #2) игнорируются

## 5. Миграция БД

**Файл:** `003_period_statistics.sql`

```sql
-- 1. Таблица статистики
CREATE TABLE period_statistics (
  id UUID PRIMARY KEY DEFAULT gen_random_uuid(),
  period_id UUID NOT NULL UNIQUE REFERENCES periods(id) ON DELETE CASCADE,
  added_to_backlog INTEGER NOT NULL,
  added_critical INTEGER NOT NULL,
  resolved_total INTEGER NOT NULL,
  resolved_critical INTEGER NOT NULL,
  in_progress INTEGER NOT NULL,
  in_testing INTEGER NOT NULL,
  locked_at TIMESTAMPTZ NOT NULL DEFAULT now(),
  created_at TIMESTAMPTZ NOT NULL DEFAULT now()
);

-- 2. Удаление устаревших полей из periods
ALTER TABLE periods DROP COLUMN IF EXISTS metrics_snapshot;
ALTER TABLE periods DROP COLUMN IF EXISTS metrics_locked_at;

-- 3. Миграция legacy статусов: 'Бэклог' → null
UPDATE tasks SET status = null WHERE status = 'Бэклог';
```

**Пояснение к миграции:**
- `metrics_snapshot` и `metrics_locked_at` переносятся из `periods` в отдельную таблицу `period_statistics` (поле `locked_at`)
- Legacy статус `'Бэклог'` конвертируется в `null` (новый начальный статус)
