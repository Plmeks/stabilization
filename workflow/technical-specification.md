# Техническая спецификация: Веб-приложение управления задачами команды

## 1. Общее описание

### Цель
Создать веб-приложение для управления задачами команды, заменяющее текущий рабочий процесс в Excel. Приложение обеспечивает полный цикл работы с задачами: от добавления QA-специалистами до завершения разработчиками, со сбором статистики по периодам.

### Технологический стек
- **Framework:** Next.js (latest, App Router)
- **Язык:** TypeScript
- **База данных:** Supabase (PostgreSQL)
- **State management:** Jotai (atom-based)
- **Стили:** Tailwind CSS
- **UI библиотека:** shadcn/ui или Ant Design (выбор разработчика)
- **Даты:** dayjs
- **Деплой:** Vercel
- **Подход:** Максимальная декомпозиция на shared-компоненты

### Доступ
- Аутентификация не требуется в первой версии
- Все пользователи имеют полный доступ ко всему функционалу
- В будущем может быть добавлена простая аутентификация

### Структура приложения
4 таба (представления):
1. Новые задачи / QA
2. Текущие задачи
3. Выполненные
4. Статистика

## 2. Список Use Cases

### UC-1: Создание периода (новый)

**Актёры:** Пользователь

**Предусловия:** Приложение открыто, пользователь на табе "Новые задачи / QA"

**Основной сценарий:**
1. Пользователь нажимает кнопку "Добавить период"
2. Система отображает модальное окно с двумя date-picker'ами (начало и конец периода)
3. Пользователь выбирает дату начала и дату конца
4. Пользователь подтверждает создание
5. Система валидирует данные (дата конца >= дата начала)
6. Система сохраняет период в БД
7. Период появляется в списке на всех табах (QA, Выполненные, Статистика)
8. Новый период отображается первым в списке (свежие вверху)

**Альтернативные сценарии:**
- 5a. Дата конца < дата начала → ошибка валидации, возврат к шагу 3
- 4a. Пользователь отменяет → модалка закрывается

**Постусловия:** Период сохранён в БД и доступен во всех табах

**Критерии приёмки:**
- Период отображается в формате "ДД.ММ - ДД.ММ.ГГГГ"
- Периоды отсортированы по дате начала (свежие вверху)
- Все периоды кроме первого (свежего) отображаются свёрнутыми
- Периоды можно сворачивать/разворачивать по клику на стрелку
- Период доступен в выпадающих списках при создании/завершении задач

### UC-2: Добавление задачи QA (новый)

**Актёры:** Пользователь

**Предусловия:** Существует хотя бы один период

**Основной сценарий:**
1. Пользователь нажимает "Добавить задачу" в табе QA
2. Система отображает модалку с полями: название задачи (текст), период (dropdown)
3. Пользователь вводит название задачи
4. Пользователь выбирает период из dropdown
5. Пользователь подтверждает
6. Система сохраняет задачу со статусом "Бэклог" (начальный статус)
7. Задача появляется под выбранным периодом в табе QA
8. Счётчик "всего" в заголовке периода обновляется

**Альтернативные сценарии:**
- 3a. Название содержит URL → система автоматически делает его кликабельной ссылкой при отображении
- 5a. Название пустое → ошибка валидации, возврат к шагу 3

**Постусловия:** Задача сохранена в БД, отображается в табе QA под выбранным периодом

**Критерии приёмки:**
- Задача отображается под правильным периодом
- URL в названии задачи — кликабельная ссылка (новая вкладка)
- Счётчик "всего" в заголовке периода увеличивается на 1
- У задачи отображается кнопка "Взять в работу"

### UC-3: Взятие задачи в работу (новый)

**Актёры:** Пользователь

**Предусловия:** Задача существует в табе QA, не взята в работу (не выделена синим)

**Основной сценарий:**
1. Пользователь нажимает "Взять в работу" напротив задачи
2. Система отображает модалку с полями (все опциональные):
   - Исполнитель (текстовое поле)
   - Приоритет (dropdown: "Авария", "Нормальный", "Некритичный")
   - Статус (dropdown: "Бэклог" выбран по умолчанию, "В работе", "В тесте", "Блокер")
3. Пользователь заполняет поля (или оставляет пустыми — можно заполнить позже)
4. Пользователь подтверждает
5. Система обновляет задачу: записывает данные, `taken_into_work_at` = now()
6. Задача в табе QA выделяется синим цветом (остаётся в списке)
7. Задача появляется в табе "Текущие задачи"
8. Счётчики обновляются

**Альтернативные сценарии:**
- 4a. Пользователь отменяет → задача не изменена

**Постусловия:** Задача взята в работу. Отображается в обоих табах (QA — синим, Текущие — в таблице).

**Критерии приёмки:**
- Все поля опциональные — можно подтвердить без заполнения
- Задача выделена синим в табе QA
- Задача отображается в табе "Текущие задачи"
- Кнопка "Взять в работу" не отображается для уже взятых задач
- Счётчики обновлены (заголовки периодов, заголовок "Текущих задач")

### UC-4: Редактирование задачи (новый)

**Актёры:** Пользователь

**Предусловия:** Задача находится в табе "Текущие задачи" или "Выполненные"

**Основной сценарий:**
1. Пользователь нажимает иконку-карандашик в столбце "Actions" напротив задачи
2. Система открывает модалку с полями: Исполнитель, Приоритет, Статус
3. Пользователь изменяет нужные поля
4. Пользователь подтверждает
5. Система сохраняет изменения в БД
6. Отображение обновляется (пересортировка при изменении приоритета)

**Альтернативные сценарии:**
- 3a. Статус изменён на "Завершена" → переход к UC-5
- 3b. В табе "Выполненные": статус изменён с "Завершена" на другой → переход к UC-6

**Постусловия:** Задача обновлена в БД и на UI

**Критерии приёмки:**
- Редактирование через модалку (НЕ inline)
- Иконка-карандашик в столбце "Actions"
- Изменения сохраняются без перезагрузки
- При изменении приоритета задача пересортируется
- Подсчёты обновляются

### UC-5: Завершение задачи (новый)

**Актёры:** Пользователь

**Предусловия:** Задача в табе "Текущие задачи", статус меняется на "Завершена"

**Основной сценарий:**
1. Пользователь меняет статус задачи на "Завершена" через модалку редактирования
2. Система отображает модалку с выбором периода для архивации
3. Dropdown содержит все периоды, самый свежий выбран по умолчанию
4. Пользователь подтверждает выбор периода
5. Система обновляет задачу: статус = "Завершена", `completed_at` = now(), `period_id` = выбранный период
6. Задача удаляется из таба "Текущие задачи"
7. Задача появляется в табе "Выполненные" под выбранным периодом
8. Счётчики обновляются

**Альтернативные сценарии:**
- 4a. Пользователь отменяет → статус остаётся прежним

**Постусловия:** Задача в "Выполненных", удалена из "Текущих"

**Критерии приёмки:**
- Задача не отображается в "Текущих задачах"
- Задача отображается в "Выполненных" под выбранным периодом
- `completed_at` заполнен
- Счётчики обновлены

### UC-6: Возврат задачи из "Выполненных" в работу (новый)

**Актёры:** Пользователь

**Предусловия:** Задача в табе "Выполненные" со статусом "Завершена"

**Основной сценарий:**

3. Задача удаляется из "Выполненных"
4. Задача появляется в "Текущих задачах"

**Постусловия:** Задача снова в активной работе

**Критерии приёмки:**
- Задача появляется в "Текущих" с правильной сортировкой
- Задача удалена из "Выполненных"
- Подсчёты корректны

### UC-7: Просмотр и фиксация статистики (новый)

**Актёры:** Пользователь

**Предусловия:** Существуют периоды и задачи

**Основной сценарий:**
1. Пользователь переходит на таб "Статистика"
2. Система отображает список периодов (свежие вверху)
3. Для каждого периода отображаются метрики:
   - "Добавлено в беклог" — кол-во задач, где `taken_into_work_at` попадает в диапазон дат периода
   - "Из них критических" — из вышеуказанных с приоритетом "Авария"
   - "Решено всего" — кол-во задач, где `completed_at` попадает в диапазон дат периода
   - "Решено критических" — из вышеуказанных с приоритетом "Авария"
   - "В работе" — из зафиксированного snapshot (или "не зафиксировано")
   - "В тесте" — из зафиксированного snapshot (или "не зафиксировано")
4. Для периодов без зафиксированных метрик отображается кнопка "Зафиксировать метрики"

**Сценарий фиксации метрик:**
1. Пользователь нажимает "Зафиксировать метрики" для периода
2. Система подсчитывает на текущий момент:
   - "В работе" = кол-во задач в табе "Текущие задачи" со статусом "В работе"
   - "В тесте" = кол-во задач в табе "Текущие задачи" со статусом "В тесте"
3. Система сохраняет snapshot в `periods.metrics_snapshot` (JSON), записывает `metrics_locked_at`
4. Кнопка исчезает, вместо неё отображаются зафиксированные значения
5. После фиксации значения "В работе" и "В тесте" не меняются

**Альтернативные сценарии:**
- Если метрики уже зафиксированы — кнопка не отображается, показываются сохранённые значения

**Постусловия:** Метрики зафиксированы и неизменны

**Критерии приёмки:**
- Метрики "Добавлено в беклог" и "Решено" подсчитываются динамически (по датам)
- Метрики "В работе" и "В тесте" фиксируются по кнопке (snapshot)
- После фиксации значения не меняются
- Кнопка "Зафиксировать метрики" видна только для незафиксированных периодов
- Периоды отсортированы свежие вверху

## 3. Нефункциональные требования

- **Производительность:** Отзывчивая работа до 1000 задач и 50+ периодов. CRUD < 500ms.
- **Совместимость:** Chrome, Firefox, Safari, Edge (последние 2 версии).
- **Адаптивность:** Desktop-first с базовой мобильной адаптацией (Tailwind responsive утилиты). На мобильном должно быть использовабельно, но без глубокой оптимизации.
- **Масштабируемость:** Легко добавить новые статусы, приоритеты, поля.
- **Переносимость БД:** Абстракция для возможной замены Supabase.

## 4. Ограничения и допущения

### Ограничения
- Тесты не пишутся (требование заказчика)
- Минимум документации
- БД — Supabase (PostgreSQL)
- Деплой — Vercel
- Без аутентификации

### Допущения
- **[Д1]** Команда 5-15 человек, real-time коллаборация не критична
- **[Д2]** Аутентификация не нужна (открытый доступ)
- **[Д3]** Supabase Realtime не нужен
- **[Д4]** Периоды — единый справочник для всех табов
- **[Д5]** При завершении задачи `period_id` обновляется на выбранный период (обычно совпадает с исходным, т.к. QA заводят задачи на период спринта)
- **[Д6]** Удаление — hard delete (не архивирование)

## 6. Anticipated Architecture Questions

1. **Роутинг:** Табы как роуты (`/qa`, `/current`, `/completed`, `/stats`) через App Router — позволяет прямые ссылки. Или один layout с client-side переключением?

2. **Синхронизация Jotai + Supabase:** Optimistic updates с откатом при ошибке, или ждать ответ сервера?

3. **Статусы в БД:** TEXT с валидацией на уровне приложения (гибкое, легко менять) — рекомендуется при 5 статусах.

4. **Подсчёт статистики:** Метрики "Добавлено в беклог" и "Решено" подсчитываются динамически (SQL или клиентский подсчёт). "В работе" и "В тесте" — из зафиксированного JSON-snapshot.

5. **UI библиотека:** shadcn/ui — легче, Tailwind-native, кастомизируемый. Ant Design — больше готового, но тяжелее.

6. **Каскадное удаление:** При удалении периода — удалять все задачи (ON DELETE CASCADE) или запрещать удаление если есть задачи?

## Схема базы данных

### Таблица `periods`
| Поле | Тип | Описание |
|------|-----|----------|
| id | UUID, PK | Уникальный идентификатор |
| start_date | DATE | Дата начала периода |
| end_date | DATE | Дата окончания периода |
| metrics_snapshot | JSONB, NULL | Зафиксированные метрики: `{"in_progress": N, "in_testing": N}` |
| metrics_locked_at | TIMESTAMP, NULL | Когда метрики были зафиксированы |
| created_at | TIMESTAMP | Дата создания записи |

### Таблица `tasks`
| Поле | Тип | Описание |
|------|-----|----------|
| id | UUID, PK | Уникальный идентификатор |
| title | TEXT, NOT NULL | Название задачи |
| period_id | UUID, FK → periods | Период (используется и в QA, и в Выполненных) |
| assignee | TEXT, NULL | Исполнитель |
| priority | TEXT, NULL | "Авария" / "Нормальный" / "Некритичный" |
| status | TEXT, NOT NULL, default "Бэклог" | Статус задачи |
| created_at | TIMESTAMP | Дата создания |
| taken_into_work_at | TIMESTAMP, NULL | Когда взята в работу |
| completed_at | TIMESTAMP, NULL | Когда завершена |

### Перечень статусов (5 штук)
1. **Бэклог** — начальный статус, по умолчанию при взятии в работу
2. **В работе** — задача в активной разработке
3. **В тесте** — задача на проверке QA
4. **Завершена** — задача выполнена
5. **Блокер** — задача заблокирована

### Приоритеты
- **Авария** — критический (считается в "крит" счётчиках)
- **Нормальный**
- **Некритичный**

### Индексы
- `tasks(period_id)` — выборка задач по периоду
- `tasks(status)` — фильтрация по статусу
- `tasks(priority)` — сортировка
- `tasks(taken_into_work_at)` — статистика
- `tasks(completed_at)` — статистика

### Логика определения "взята в работу"
- Задача считается "взятой в работу" если `taken_into_work_at IS NOT NULL`
- В табе QA такая задача выделяется синим
- В табе "Текущие задачи" отображаются задачи с `taken_into_work_at IS NOT NULL` и `status != 'Завершена'`
- В табе "Выполненные" отображаются задачи с `status = 'Завершена'`
