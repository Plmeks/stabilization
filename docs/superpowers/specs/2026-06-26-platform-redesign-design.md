# Редизайн платформы Stabana + Enter-подтверждение + выпадашка «Исполнитель»

Дата: 2026-06-26

## Цель

Перенести визуальный язык PDF-отчёта (`ReportDocument.tsx`) на всё приложение,
включая мобайл. Плюс две функциональные задачи:

1. Enter подтверждает действие в модалках создания/редактирования (не удаление/откат).
2. Поле «Исполнитель» — выпадашка из управляемого списка (＋ сохранить / ✕ удалить),
   с очисткой поля крестиком.

Тема: **светлая** с небольшими тёмными акцентами. Полноценная dark-тема не нужна
(сейчас `.dark` определена в CSS, но нигде не включается).

## Часть 1 — Дизайн-система

Источник истины — палитра из `src/components/report/ReportDocument.tsx`.

акцент-токены. Тёмную тему (`.dark`) не трогаем/удаляем как неиспользуемую — приложение
светлое.

### Типографика
- Geist (как сейчас).
- `font-variant-numeric: tabular-nums` для чисел (таблицы, счётчики, статистика).
- Eyebrow-подписи разделов: UPPERCASE, letter-spacing ~0.12em, ~11px, slate-400, semibold.

### Мотивы
- **KPI / стат-карточки**: цветная верхняя кромка 3px (как в отчёте).
- **Лента стабильности**: тонкая 4-цветная полоска (success/danger/warn/wip) — сигнатура.
  Размещение: под шапкой навигации (во всю ширину).
- **Маленькие тёмные акценты**: счётчики-чипы, активная вкладка, основные кнопки — slate-900.

### Шапка + логотип

## Часть 2 — Enter подтверждает в модалках

`ModalWrapper` получает необязательный проп `onSubmit?: () => void`.
Когда он задан — тело + футер оборачиваются в `<form onSubmit={...}>`, основная кнопка
футера получает `type="submit"`, кнопка «Отмена» — `type="button"`.

Нативное поведение формы: Enter в однострочном `<input>` сабмитит; Enter в `<textarea>`
вставляет перенос строки (так и нужно). `preventDefault` в обработчике.

Применить (передать `onSubmit`, основная кнопка → submit):
`AddTaskModal`, `EditTaskModal`, `CreatePeriodModal`, `TakeIntoWorkModal`,
`CompleteTaskModal`, `CommentModal`, `EditQATaskModal`, `EditMetricsModal`.

**НЕ применять**: `ConfirmDialog` (удаление, возврат в QA / откат) — Enter не подтверждает.

## Часть 3 — Выпадашка «Исполнитель»

### Хранилище
- Таблица Supabase `assignees`: `id` (uuid, pk), `name` (text, unique), `created_at`.
- Миграция в `supabase/`.
- DAL (`src/lib/supabase/dal.ts`): `fetchAssignees`, `createAssignee(name)`,
  `deleteAssignee(id)`.
- Атом (`src/atoms/assigneesAtom.ts`): `assigneesAtom`, `fetchAssigneesAtom`,
  `createAssigneeAtom`, `deleteAssigneeAtom` (оптимистично, как в tasksAtom).
- Загрузка списка в `DataLoader`.

### Компонент `src/components/shared/AssigneeCombobox.tsx`
Props: `value: string`, `onChange: (v: string) => void`, `disabled?`, `id?`.

Поведение:
- Текстовое поле + Popover-список сохранённых исполнителей.
- Ввод фильтрует список. Если введённого имени нет в списке — справа в поле иконка
  **＋** (сохранить имя в список через `createAssigneeAtom`).
- Каждый элемент списка: имя + маленький **✕** справа — удалить из списка
  (`deleteAssigneeAtom`).
- Маленький **✕** в самом поле (когда есть значение) — очистить значение (паттерн как
  в `SearchInput`).
- Клик по элементу списка — подставляет имя в поле (`onChange`).

Использование: `TakeIntoWorkModal`, `EditTaskModal` (замена `<Input>` исполнителя).
